4）执行要点（请在方案与代码中体现）

4.1 需求澄清与任务拆解
- 根据项目上下文，将目标任务拆解为明确的子任务和 Agent 能力模块，形成**SOP / 里程碑列表**，例如：
  - 识别用户问题 → 检索上下文 → 规划工具调用序列 → 执行工具 → 汇总结果 → 生成可交付产出。
- 明确输入 / 输出边界：
  - 输入：用户自然语言 / 结构化请求字段 / 上游系统回调的 payload 格式；
  - 输出：结构化结果（JSON）、用户可读说明、需要写回的实体（工单/配置/记录等）。
- 明确定义每个关键阶段的**验收标准**，例如：
  - 诊断 Agent：召回的关键日志覆盖率 ≥ X%，错误定位正确率 ≥ Y%；
  - 执行 Agent：执行成功率 ≥ Z%，失败时能返回可复现信息。
- 在回答中给出任务分解表（任务 → 输入 → 输出 → 所需工具/数据 → 验收标准）。

4.2 Agent 架构与技术选型
- 明确整体架构形态：
  - 单体 Agent / 多 Agent 编排 / 分层架构（入口 Orchestrator + 领域子 Agent + 工具适配层）；
  - 是否有 Planner / Executor / Critic / Memory 等角色划分。
- 对 Planner/Executor/Memory/Tooling 的设计给出清晰职责：
  - Planner：负责任务拆解、步骤规划、工具选择与调用顺序；
  - Executor：按照计划执行具体工具调用与状态更新；
  - Memory：维护会话级 / 任务级记忆，负责读写向量库、KV 或文件系统；
  - Tooling：封装真实后端服务、第三方接口、内部系统（通过 MCP / Nacos / HTTP / gRPC 等）。
- 明确流程编排方式：
  - 使用工作流引擎（如内部 workflow、DAG、状态机）或代码内显式状态机；
  - 是否使用事件总线（EventBus）和队列（如 Kafka / MQ）处理异步任务和重试。
- 对并发与队列设计给出建议：
  - 控制并发度与队列长度，避免对下游服务产生突发压力；
  - 对长耗时任务设计异步回调或轮询机制（避免同步阻塞）。
- 明确技术选型理由（如：为什么使用某个框架/SDK/工作流引擎），避免“仅罗列名称”。

4.3 模型与推理策略
- 明确模型选择：
  - 指定主模型（如内部模型代号、gpt-4.1 等）以及备选模型；
  - 对不同子任务可采用不同模型（如推理型 vs 生成型）。
- 设计推理参数与模式：
  - 给出默认温度、top-p、max_tokens、停用词等推荐值；
  - 对**Deterministic（确定性）** vs **Creative（创意型）**场景分别给出设置建议，例如：
    - 工单执行/调度：温度接近 0、top-p 较低，保证可重复性；
    - 文案生成：允许更高温度与多样性。
- 说明思维链 / 计划生成策略：
  - 是否使用显式 CoT / ReAct / Plan-and-Execute；
  - 如何控制思维链长度，避免冗长与 token 浪费；
  - 在工具调用前是否要求模型先输出结构化的计划（JSON 或步骤列表）。
- 指出异常场景处理策略：
  - 模型输出不符合结构时的修正与重试；
  - 模型不确定时如何引导其标注不确定性并请求更多上下文。

4.4 工具 / 技能接入与治理
- 列出需要接入的工具/服务与形式：
  - MCP 工具（遵循 MCP 协议的工具服务器）；
  - Nacos 注册的微服务（Kitex / HTTP / gRPC 等）；
  - 其他内部平台（搜索、日志、监控、配置等）。
- 说明工具接入方式：
  - 鉴权方式（token / AKSK / 内网权限），以及如何安全注入（环境变量 / KMS）；
  - 速率限制与配额控制（QPS 限制、调用配额、熔断阈值）。
- 设计失败重试 / 回退策略：
  - 哪些错误可重试（网络抖动、超时等），重试次数与退避策略；
  - 哪些错误应立即回退并提示（鉴权失败、权限不足、参数非法等）；
  - 当关键工具不可用时的回退方案（例如：退化为只读诊断、不自动执行变更）。
- 明确工具选择策略：
  - 当多个工具功能相近时，如何根据任务参数、环境、成本选择；
  - 是否允许模型“猜测”工具用法，还是强制走预定义模板。

4.5 记忆与检索
- 区分短期上下文与长期存储：
  - 短期：当前对话 / 当前任务的多轮状态；
  - 长期：用户画像、历史工单、历史执行结果、场景知识库等。
- 设计记忆存储方案：
  - 向量检索：说明使用的向量库（内部向量服务 / 数据结构）、索引粒度（文档/段落/记录）、维度与相似度度量；
  - 文件系统：如何组织日志、配置、导出报告等文件；
  - KV / 数据库：存储会话状态、任务状态、回放轨迹等。
- 控制信息过载：
  - 对检索到的内容进行聚类、摘要与压缩，限制注入模型的 token 数量；
  - 为每条注入上下文附加来源标识与链接，便于解释与回溯。
- 设计“笔记与压缩”机制：
  - 对长对话或长任务，在关键节点生成摘要笔记存入长期记忆；
  - 定期对记忆进行清理与归档，避免无边界膨胀。

4.6 安全与合规
- 明确权限边界：
  - 列出 Agent 允许调用的工具/接口，以及禁止调用或需额外确认的操作（如删库、批量封禁、敏感配置变更）；
  - 如涉及多租户或账号权限，说明如何在调用工具时附带并校验租户/用户身份。
- PII 与敏感信息处理：
  - 对用户标识、手机号、邮箱、日志内容等进行脱敏或部分掩码；
  - 明确哪些字段不得写入长期记忆或外部系统。
- 审计与可回溯：
  - 为每次 Agent 决策与工具调用记录结构化审计日志（包含 log_id、用户、时间、工具名、参数、结果摘要）；
  - 保证可以重放关键决策路径（例如保存规划结果与实际执行步骤）。
- 数据最小化：
  - 仅在必要范围内收集与传递数据；
  - 对模型与工具调用的输入进行裁剪和筛选，避免把整个大对象随意传入。

4.7 可观测性与日志
- 设计任务 Trace：
  - 每个任务生成唯一 trace_id / log_id，贯穿入口请求、Agent 规划、工具调用、模型推理等全链路；
  - 使用结构化日志记录每个步骤的开始/结束时间、耗时与状态。
- 日志设计：
  - 约定统一的日志字段（psm、env、lane、user_id、task_id、agent_name、tool_name、error_code、latency_ms 等）；
  - 日志级别划分（INFO/DEBUG/WARN/ERROR），避免过度 DEBUG 造成噪音与成本。
- 指标与告警：
  - 定义关键指标：任务成功率、平均/分位响应时间、工具调用失败率、重试次数、模型 token 消耗等；
  - 为关键错误与异常情况配置告警（如工具持续失败、评测得分连续下降）。
- 问题自诊断输出：
  - 当 Agent 检测到异常时，在用户可见输出中附上简洁的自诊断总结（例如：哪个工具失败、是否自动重试、建议提供哪些信息）。

4.8 评测与质量保障

  - 指定评测框架（例如：基于 Benchmark/GAIA 式的评测任务），说明指标与评分方法。
- 在线评测 / A/B：

  - 定义观测窗口与统计指标（成功率、人工介入比例、用户满意度、决策延迟）。
- 通过阈值与回归套件：

- 在回答中提供**具体的评测用例示例**（包括输入、期望输出、判分标准）。

4.9 提示工程与输出规范

  - 规定回答的语气与风格（例如：面向客服的解释 vs 面向工程师的诊断报告）。
- 结构化输出：

  - 对于人类阅读的报告，要求结构化（标题、小节、清单），避免长段落堆砌。
- 错误处理与自我纠错：
  - 当检测到前后输出不一致、不符合 schema 或工具报错时，主动触发自检与重试；
  - 在输出中标注“已重试”与“仍然失败”的关键信息，并给出人工介入建议。
- 重试策略：
  - 对模型生成失败（如 API 超时）与工具调用失败分别设计重试次数与间隔；
  - 注意避免无限循环，设置最大尝试次数与 fallback 行为。
